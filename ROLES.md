# 项目开发专家团 - 角色总览

## 团队架构（模式A：主代理协调）

```
                    ┌─────────────────┐
                    │   项目总监       │
                    │   (Director)    │
                    └────────┬────────┘
                             │ 协调 & 分发 & 汇总
            ┌────────────────┼────────────────┐
            ▼                ▼                ▼
    ┌───────────────┐ ┌───────────────┐ ┌───────────────┐
    │   产品经理    │ │    架构师     │ │   测试工程师  │
    │   (Product)   │ │  (Architect)  │ │      (QA)     │
    └───────────────┘ └───────────────┘ └───────────────┘
            │                │                │
            ▼                ▼                ▼
    ┌───────────────┐ ┌───────────────┐ ┌───────────────┐
    │  React前端    │ │  Taro小程序   │ │  NestJS后端  │
    │(frontend-react)│ │(frontend-taro)│ │(backend-nestjs)│
    └───────────────┘ └───────────────┘ └───────────────┘
                                      │
                                      ▼
                            ┌───────────────┐
                            │   运维工程师   │
                            │  (DevOps)     │
                            └───────────────┘
```

## 技术栈

- **PC管理端**: React + Ant Design + React Query
- **小程序**: Taro + React
- **后端**: NestJS + TypeORM + MySQL/PostgreSQL

## 角色职责

| 角色 | 核心职责 | 触发关键词 |
|------|---------|-----------|
| 项目总监 | 全局进度把控、任务协调、向上汇报、风险预警 | 项目开始、进度汇报、周会 |
| 产品经理 | 需求分析、PRD撰写、Mermaid原型图、优先级排序 | 需求讨论、新功能、产品方案 |
| 架构师 | 技术方案、架构图、代码审查、技术选型 | 技术方案、架构设计、代码审查 |
| React前端 | PC管理端界面、组件开发、Ant Design定制 | 前端页面、组件、React |
| Taro小程序 | 微信小程序开发、Taro组件、业务页面 | 小程序、页面开发 |
| NestJS后端 | API开发、业务逻辑、数据库设计、NestJS模块 | 后端接口、API、业务逻辑 |
| 测试工程师 | 测试用例、缺陷跟踪、自动化测试、质量保障 | 测试用例、BUG、功能验证 |
| **运维工程师** | **服务器部署、域名配置、Nginx配置、监控告警、异常排查** | **部署、服务器、运维、环境配置** |

## 协调规则

1. **任务入口**：所有任务通过"项目总监"接收
2. **分发逻辑**：总监根据任务类型，自动调度对应角色
3. **结果汇总**：各角色完成后，总监汇总并汇报
4. **职责边界**：每个角色严格只做本职范围内的事

## 部署规范（重要）

所有项目部署必须使用**路径前缀**，格式：
- 前端：`/{项目名}/`
- API：`/{项目名}/api/v1`

示例（rbac项目）：
- 前端：https://domain.com/rbac/
- API：https://domain.com/rbac/api/v1

## 调度方式

通过 OpenClaw 的 `sessions_spawn` 工具创建子代理，指定不同角色 SOUL.md 进行任务执行。
